## рҹҡ§ The Problem

Currently, gig workers have **zero income protection** against these uncontrollable events. When disruptions occur, they bear the full financial loss with no safety net.

| Reality | Impact |

| Daily income loss per disruption | вӮ№300 вҖ“ вӮ№800 |
| Monthly income at risk | 20вҖ“30% |
| Available insurance products | **Zero** |
| Compensation from platform | **None** |
| Backup savings buffer | Near zero |

> **The invisible risk zone:** Severe weather В· Heavy pollution В· Curfews & lockdowns В· Road blockages вҖ” *No work. No pay. No safety net.*

### Phase-by-Phase Detail

**Phase 1 вҖ” Onboarding:** Handles KYC via Aadhaar OTP + DigiLocker, AI-driven risk scoring, and NLP-guided coverage selection so workers understand exactly what they're buying.

**Phase 2 вҖ” Policy Config:** AI defines precise trigger thresholds (e.g. rainfall вүҘ 94mm in 3hrs, AQI вүҘ 300), dynamically prices the premium using historical data, and runs back-test simulations showing what payouts would have looked like in past events.

**Phase 3 вҖ” Issuance:** Encodes policy terms into a smart contract or signed digital ledger, eliminating ambiguity at claim time.

**Phase 4 вҖ” Monitoring:** Data oracles (weather APIs, AQI feeds, civic disruption data) stream into an ML anomaly detection engine that watches for threshold breaches 24/7.

**Phase 5 вҖ” Claims:** Fully automated. Trigger fires вҶ’ system self-files the claim вҶ’ cross-validates oracle data from multiple sources вҶ’ fraud check вҶ’ computes income deviation payout. No action needed from the worker.

**Phase 6 вҖ” Payout:** Funds delivered within minutes. Full audit trail surfaces exactly why a payout was (or wasn't) triggered, maintaining complete trust.

## вҡЎ Parametric Triggers

All triggers are hyperlocal вҖ” validated at the worker's specific delivery zone, not city-wide alerts.

| # | Trigger | Data Source | Threshold | Notes |
|---|---------|-------------|-----------|-------|
| рҹҢ§пёҸ 1 | **Heavy Rain / Floods** | IMD + OpenWeatherMap | Rainfall вүҘ 94mm in 3hrs / Red Alert issued | Core Chennai/Mumbai monsoon risk |
| рҹҢЎпёҸ 2 | **Extreme Heat** | IMD + Open-Meteo | Temperature вүҘ 43В°C or official advisory | Summer months, outdoor suspension |
| рҹҢ«пёҸ 3 | **High Pollution (AQI)** | CPCB AQI Feeds + OpenAQ | AQI вүҘ 300 (Hazardous) | Delhi, Bengaluru Diwali season |
| рҹҡ§ 4 | **Curfew / Lockdown** | Civic Disruption APIs | Section 144 or mobility index < 20% | Political events, protests |
| рҹҢҠ 5 | **Road Blockage / Floods** | Traffic API + IMD | Zone roads impassable > 2hrs | Flood-zone specific routing |

### Income Deviation Formula

```
Actual Payout = (30-Day Baseline Daily Income) Г— (Hours Lost / Normal Active Hours) Г— Severity Factor

Where:
  30-Day Baseline = Rolling average of worker's actual daily earnings (last 30 days)
  Hours Lost      = Active working hours disrupted (platform-verified)
  Severity Factor = 0.5вҖ“1.0 based on disruption intensity

Example вҖ” Ravi, Chennai Flood:
  Baseline daily income : вӮ№700
  Hours lost            : 4 hrs out of 8hr shift (50%)
  Severity factor       : 0.90 (Red Alert = extreme)
  Payout = вӮ№700 Г— 0.50 Г— 0.90 = вӮ№315 вҶ’ rounded to вӮ№320 credited via UPI
```

### 1. Predictive Risk Model

The system predicts disruption probability and dynamically prices each worker's weekly premium using:

- Zone-specific flood/heat/AQI historical data (IMD + CPCB, 2019вҖ“2024)
- Seasonal disruption patterns (monsoon multipliers, Diwali AQI spikes)
- Worker platform tenure and active hours consistency
- Location risk index (flood-prone zones, high-pollution corridors)

### 2. Behavioral Fraud Detection (Isolation Forest)

Detects anomalies using a **30-day rolling behavioral baseline** per worker:

| Fraud Vector | Detection Method |
|---|---|
| GPS spoofing | Device authentication + IP check + cell tower triangulation |
| Impossible location shifts | Geospatial velocity check (>50km/hr = flag) |
| Fake disruption claims | Multi-source oracle cross-validation (3 independent APIs) |
| Abnormal claim frequency | Velocity rule: max 2 claims per 7-day window before review |
| Income inflation | Cross-check declared income vs. platform-reported 30-day baseline |
| Inconsistent activity patterns | Isolation Forest anomaly score on behavioral time-series |

### 3. Real-Time Income Deviation Engine

Unlike traditional parametric insurance (which pays a fixed amount per event), Income Shield AI computes **exact income deviation** against each worker's personal baseline:

- Workers with higher baselines get proportionally higher payouts
- Workers who were not actively logged in at disruption time receive reduced payouts
- Partial disruptions (2 hours out of 8) get partial payouts вҖ” not binary full/nothing

## рҹ’° Weekly Premium Model

### Pricing Philosophy

> "Earn weekly. Pay weekly. Stay protected weekly."

Our pricing model is designed specifically for gig workers whose income is **short-term, variable, and location-dependent**. Instead of monthly or yearly plans, we use a dynamic weekly premium that adapts to real-time risk, remains affordable, and aligns with weekly earning cycles.

### AI-Based Pricing Formula

```
Weekly Premium = Base Rate + (Location Risk Index Г— 0.35) + (Disruption Probability Г— Coverage Amount Г— 0.08)

Components:
  Base Rate             = Standard minimum coverage cost (вӮ№20 baseline)
  Location Risk Index   = Area-specific risk score (0.0вҖ“1.0)
                          Flood-prone zones В· High AQI corridors В· Traffic congestion
  Disruption Probability = AI-predicted probability for next 7 days
                          Using weather forecasts + historical data + seasonal patterns

Example вҖ” Ravi, Chennai, Monsoon Season:
  Base Rate        = вӮ№20
  Location Index   = 0.72 (flood-prone zone)
  Disruption Prob  = 0.45
  Coverage Amount  = вӮ№3,000

  Weekly Premium   = вӮ№20 + (0.72 Г— 0.35 Г— 100) + (0.45 Г— вӮ№3,000 Г— 0.08)
                   = вӮ№20 + вӮ№25.20 + вӮ№108
                   = вӮ№153.20 / week  (~5% of weekly income)
```

### Premium Tiers (Indicative)

| Zone Risk | Location Type | Weekly Premium | Weekly Coverage | % of Income |
|---|---|---|---|---|
| рҹҹў Low | Stable zone, low disruption history | вӮ№29/week | вӮ№1,800/week | ~1.1% |
| рҹҹЎ Moderate | Seasonal exposure, some disruption risk | вӮ№49/week | вӮ№3,000/week | ~1.4% |
| рҹ”ҙ High | Flood-prone / AQI zone / bandh-frequent | вӮ№79/week | вӮ№5,000/week | ~1.8% |

> Premiums are **recalculated every Monday** using fresh AI inference вҖ” zone risk, seasonal forecast, worker tenure, and 30-day earning history.

## рҹӣЎпёҸ Adversarial Defense & Anti-Spoofing

Income Shield AI uses a **multi-layer validation architecture** to protect against GPS spoofing, fake data injection, and manipulation attempts.

### Defense Layers

```
LAYER 1 вҖ” DEVICE AUTHENTICATION
  Device fingerprint В· App attestation В· Jailbreak/root detection

LAYER 2 вҖ” LOCATION VERIFICATION
  GPS coordinates + Cell tower triangulation + IP geolocation
  Velocity check: location shift > 50km/hr = impossible movement flag

LAYER 3 вҖ” DATA ORACLE CROSS-VALIDATION
  3 independent API sources must agree on disruption event
  IMD + OpenWeatherMap + CPCB must all confirm before trigger fires

LAYER 4 вҖ” BEHAVIORAL ANOMALY DETECTION
  Isolation Forest on 30-day rolling baseline
  Flags: unusual claim velocity, income deviation outliers

LAYER 5 вҖ” AUDIT TRAIL
  All trigger decisions logged immutably
  Workers see exactly why a payout was or was not issued
  Insurer dashboard shows full SHAP explainability
```

### Anti-Spoofing Strategy

- Suspicious claims are **blocked or delayed** until multi-source validation completes
- Workers can see exactly why their claim was held вҖ” full transparency
- Behavioral monitoring continues across rolling windows вҖ” patterns identified over time, not just per-event
- All actions stored in immutable secure audit logs for regulatory compliance

## рҹ”ҙ What Income Shield AI Strictly Excludes

The problem statement has a hard constraint вҖ” we never touch these:

| вқҢ Excluded | Reason |
|------------|--------|
| Vehicle repairs or maintenance | Personal mechanical failure вҖ” not an external parametric trigger |
| Health or medical expenses | Out of coverage scope |
| Life or accident insurance | Separate regulated product category |
| Platform commission changes | Business policy decision, not an environmental disruption |
| Worker's voluntary absence | Personal choice вҖ” not an uncontrollable external event |

The system only pays when an **external, measurable, zone-level event** causes income loss that the worker had absolutely no control over.

## рҹҢҹ Why Income Shield AI Wins

| Feature | Why It Matters |
|---------|----------------|
| **Income Deviation Model** | Unlike fixed-amount parametric insurance, we compute exact personal income loss against each worker's 30-day baseline вҖ” Ravi gets paid for what *he* actually lost |
| **Hyperlocal Triggers** | Zone-level polygon validation вҖ” not city-wide alerts. Ravi's Adyar zone must actually be affected |
| **30-Day Rolling Baseline** | Fraud engine learns each worker's normal behavior вҖ” unusual patterns over time, not just per-event anomalies |
| **Dual-Layer AI** | Predictive risk pricing + behavioral fraud detection running simultaneously |
| **Multi-Source Oracle** | 3 independent APIs must agree before a trigger fires вҖ” eliminates single-point-of-failure false payouts |
| **Immutable Audit Trail** | Every trigger decision logged with full SHAP explainability вҖ” workers see exactly why payout was or wasn't issued |
| **Monday Auto-Renewal** | Weekly premium aligns perfectly with Swiggy/Zomato payment cycles вҖ” zero friction |
